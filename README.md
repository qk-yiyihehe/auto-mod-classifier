# 自动筛选模组分类器 2.00

一个面向 Windows 的 Minecraft 模组筛选与自动建服工具。

这次把 GitHub 对外发布版本名整理为 `2.00`，用来和早期 `1.0` 半成品区分。当前仓库对应的是这套稳定代码快照，功能上已经包含加入“一键制作服务端”之后的完整能力。

## 2.00 包含的核心能力

- `Mod筛选模式`
  - 选择 `mods` 目录后，自动分类为 `服务端保留 / 纯客户端 / 待人工确认`
  - 支持 `Fabric / Quilt / Forge / NeoForge`
  - 支持本地元数据判断、`Modrinth` 联网判断、`MC百科` 兜底判断
  - 支持可选 `2次筛选`，只重试首轮 `unknown`
  - 会生成 `JSON / CSV / TXT` 报告，并单独标记损坏的 Jar
- `一键制作服务端模式`
  - 选择整个客户端实例目录和新的空输出目录
  - 自动识别 `Minecraft 版本 + 加载器类型 + 精确加载器版本`
  - 支持 `Fabric / Forge / NeoForge` 自动建服
  - 自动下载官方安装器并安装服务端
  - 自动筛选客户端 `mods` 并复制到服务端
  - 自动复制常见配置目录，并提供人工勾选核查
  - 自动生成启动脚本
  - 自动首启生成配置、写入 `eula=true`、改为 `online-mode=false`
  - 自动进行第二次启动验证

## 自从加入服务端制作后的主要新增功能

- 主界面从单模式扩展为双模式：
  - `Mod筛选模式`
  - `一键制作服务端模式`
- 支持识别 `.minecraft` 根目录，也支持 PCL 常见的版本根目录直开
- 自动匹配 Java 版本，并区分不同 Minecraft 版本需求
- 服务端制作流程加入两个核查页：
  - `Mod复制核查`
  - `配置目录复制核查`
- 模组和目录核查列表支持：
  - 鼠标滚轮滚动
  - 左键点击文件名直接复制
- 版本候选改为自然排序，避免 `1.20.10` 和 `1.20.2` 错序
- Forge / NeoForge 启动链路改成保留官方 `run.bat`
- `win_args.txt` 优先按标准路径精确定位，不再全目录乱扫
- 一键建服失败时也会尽量保留：
  - 模组筛选报告
  - 目录复制摘要
  - 制作日志
- MC百科 请求加入同 key 并发去重，减少重复联网和 IP 压力
- 联网分类支持 `2次筛选`，只对首轮 `unknown` 模组重试
- `lib / library / api / mod` 这类后缀会放宽匹配，降低库模组误判
- 中文 CSV 表头、损坏 Jar 标记、最终 `unknown` 重点日志输出都已补齐

## 直接运行源码

需要本机已安装 Python 3。

```powershell
pythonw .\自动筛选模组分类器.pyw
```

或者直接双击：

```text
启动自动筛选模组分类器.bat
```

## 重新打包

仓库内已提供：

```text
打包发布.bat
```

它会自动寻找当前目录下的入口 `pyw` 和图标 `ico`，并调用 PyInstaller 生成带图标的单文件 exe。

## 使用说明

### Mod筛选模式

- 输入一个 `mods` 目录
- 程序先读取 jar 元数据
- 本地能判断的先本地判断
- 本地判不准再查 `Modrinth`
- 仍不确定时可继续用 `MC百科` 兜底
- 最终输出分类报告，并把 `纯客户端` 与 `待人工确认` 分目录整理

### 一键制作服务端模式

- 输入整个客户端实例目录，不是单独的 `mods` 目录
- 输出目录必须是新的空目录
- 支持自动识别客户端版本清单与加载器版本
- 自动建服当前支持：
  - `Fabric`
  - `Forge`
  - `NeoForge`
- `Quilt` 当前只保留模组筛选识别，不进入自动建服流程

## 注意事项

- 模组联网分类依赖网络，速度会受 `Modrinth / MC百科` 请求情况影响
- 一键建服不会修改客户端源目录
- 如果版本清单中无法可靠识别出精确加载器版本，程序会直接报错，不做模糊推断
- 当前仓库公开发布名使用 `2.00`，但源码目录里仍保留原有的阶段性产物命名
